01

Vision & goals

DistributedLab is inspired by two landmark books in the field — Building Microservices and Designing Data-Intensive Applications — which share a common insight: you cannot truly internalize distributed systems concepts by reading alone. Partial failure, replication lag, and service coupling only become viscerally real when you break them in a running system.

This platform is a self-hosted "living laboratory" — a Docker Compose environment bundled with a web control panel — that lets any reader take these concepts off the page and into a real (simulated) system.

Primary audience

Engineers exploring distributed systems and microservices who want to experiment with architectural patterns in a safe, isolated environment.

Community goal

An open-source repo that the wider developer community can contribute new experiments, lab scenarios, and service templates to.

Core promise

One docker compose up launches a full multi-service environment plus a control panel. No cloud account required.

Out of scope (v1)

Multi-user auth, cloud deployment, production workload handling. The sandbox is intentionally local and educational.

02

Lab modules

The platform is structured as five independent but interconnected lab modules. Each covers a cluster of concepts central to distributed systems and microservice architecture.

Migration & decomposition lab

Strangler Fig, Parallel Run, Feature Toggles

Module 01
  • Majestic Monolith template A working "MusicCorp"-style Spring Boot monolith users can clone and incrementally decompose. All data in a single PostgreSQL schema to start.
  • Strangler Fig proxy A configurable HTTP proxy (Nginx + Lua, or Envoy) where learners redirect specific routes from the monolith to a new microservice via the control panel — no monolith code change required.
  • Parallel Run comparator Fire the same request at both the monolith and the extracted service, diff the JSON responses side-by-side in the UI to build confidence before cutover.
  • Feature toggle dashboard Runtime control panel to flip feature flags between old and new implementations without redeployment. Backed by Redis.

Distributed data & consistency sandbox

CAP theorem, replication lag, event sourcing

Module 02
  • CAP theorem visualizer A two-node key-value store where the learner manually "cuts" the network link between nodes via the UI. A live dashboard shows whether the system chooses Consistency (reject writes) or Availability (allow divergent writes).
  • Replication lag simulator Configurable async leader-follower replication (PostgreSQL logical replication). A slider controls artificial lag. Read-your-writes and monotonic reads violations are surfaced visually in real time.
  • Data model "A/B" explorer Implement the same social-network domain in three storage backends — Relational (PostgreSQL), Document (MongoDB), Graph (Neo4j). Run the same business queries against each and compare latency and schema complexity.
  • CDC & event sourcing pipeline Debezium captures change events from PostgreSQL, streams to Kafka, which updates an Elasticsearch read model. Learners can pause, replay, and corrupt events to see downstream effects.

Resiliency & chaos console

Chaos engineering, circuit breakers, bulkheads

Module 03
  • Chaos engineering panel A control panel to inject faults into running containers: kill a service, add network latency (tc netem), drop packets, or exhaust memory. Powered by Chaos Toolkit / Toxiproxy.
  • Stability pattern playground Pre-wired circuit breaker (Resilience4j) and bulkhead templates. Learners configure thresholds and watch the breaker open/close in real time as a downstream service degrades.
  • Cascading failure visualizer A live dependency graph that highlights which services are degraded or failed, showing whether bulkheads successfully contain the blast radius.

Workflow & communication explorer

Sagas, idempotency, messaging patterns

Module 04
  • Saga visualizer — orchestrated & choreographed The same order-fulfillment process implemented twice: once with a central orchestrator and once with pure choreography (events). A real-time map shows messages flowing between services and where coupling occurs.
  • Idempotency tester Replay the same POST request N times and verify that side effects (e.g. payment) occur exactly once. An Idempotency-Key mechanism is pre-wired; learners can disable it and observe double-processing.
  • Communication style lab Implement the same service interaction using REST, gRPC, and Kafka. The UI shows latency, throughput, and temporal coupling differences side-by-side under load.
  • Consumer-driven contract tester (Pact) A built-in Pact Broker. Learners define consumer contracts, run provider verification, and deliberately introduce breaking changes to see CI fail fast.

Observability suite — the "paved road"

Metrics, tracing, service registry

Module 05
Build the observability foundation first. This module ships pre-configured and is used by all other modules.
  • Unified metrics & log aggregator Prometheus scrapes all services automatically. Grafana dashboards are pre-built. Loki/Promtail collects structured logs. Zero configuration for learners.
  • Distributed tracing dashboard (Jaeger) Correlation IDs propagated automatically via OpenTelemetry. The control panel lets learners follow a single request across N service hops and identify the slowest span.
  • Humane service registry A lightweight Backstage-inspired catalog that lists each sandbox service with: owner, health status, API docs link, current version, and upstream/downstream dependencies.
03

System architecture

The entire platform runs locally via docker compose up. A Next.js control panel (the "DistributedLab UI") orchestrates experiments by calling a thin Spring Boot "Lab API" that manipulates containers and injects faults via the Docker socket and Toxiproxy.

High-level component layout
Learner browserDistributedLab UI (Next.js)
↕
Control layerLab API (Spring Boot) · REST + WebSocket
↕
Fault injectionToxiproxy · Docker API
Sandbox servicesMonolith · Microservices
ObservabilityPrometheus · Jaeger · Loki
↕
StoragePostgreSQL · MongoDB · Neo4j · Redis · Kafka
Proxy layerEnvoy · Nginx (Strangler Fig)
Docker Compose network WebSocket for real-time events Docker socket (fault injection)
The Docker socket is mounted read-write into the Lab API container for fault injection. The sandbox should never run on a production host or with untrusted users.

Repository structure

A monorepo makes it easy for the community to clone the entire lab at once. Folder boundaries maintain clear service separation.

Build roadmap

Phase 1 — Foundation (weeks 1–3)

Observability-first skeleton

  • Docker Compose with Prometheus, Grafana, Jaeger, Loki
  • Control panel shell (Next.js, dark/light mode, sidebar nav)
  • Lab API with Docker socket integration (start/stop containers)
  • Monolith skeleton (MusicCorp — catalog, orders, payments)
  • Service registry (Backstage lite)
Phase 2 — Module 01 (weeks 4–6)

Migration lab

  • Envoy Strangler Fig proxy with UI route-redirect controls
  • Parallel Run comparator (diff viewer)
  • Feature toggle dashboard (Redis-backed)
  • Extract catalog-service from monolith as worked example
Phase 3 — Module 02 (weeks 7–10)

Data & consistency sandbox

  • PostgreSQL leader-follower replication + lag slider
  • CAP partition simulator (two-node KV store, iptables)
  • Multi-model explorer (PostgreSQL / MongoDB / Neo4j)
  • Debezium CDC → Kafka → Elasticsearch pipeline
Phase 4 — Modules 03 & 04 (weeks 11–15)

Chaos, workflows & communication

  • Toxiproxy fault injection panel
  • Resilience4j circuit breaker demo service
  • Saga visualizer (orchestrated + choreographed)
  • Pact Broker integration and CDC tester
  • Communication style lab (REST vs gRPC vs Kafka)
Phase 5 — Open source launch (week 16+)

Community & polish

  • Contributor guide, issue templates, good-first-issue labels
  • Lab scenario plugin API (community can add new labs)
  • GitHub Actions CI (test + compose health check)
  • Demo video + docs site (Docusaurus)
07

Design principles

Observability first

Every experiment is only as valuable as the signal it produces. The observability stack ships pre-configured and is never optional.

One command to start

docker compose up must be the only prerequisite. No cloud account, no manual config, no database seed scripts to run by hand.

Failure is the feature

The platform should make it trivially easy to break things. Chaos and fault injection are first-class UI elements, not afterthoughts.

Paved road for contributors

A _template skeleton service has logging, health checks, and OpenTelemetry pre-wired. New lab scenarios follow a defined plugin contract.

Concept-first naming

UI labels always reference the originating concept (e.g. "Strangler Fig"), not the implementation tool (e.g. "Envoy route config"), to reinforce learning.

Intellectual honesty

The repo README credits both inspiring books. No book content is reproduced verbatim. All code is original implementation of the described patterns.
